# Архитектура первой версии

## Решение

Проект сделан отдельным локальным приложением Bitrix24 в папке `/local/purchase_requests/`.

Bitrix24 отвечает за:

- вход пользователя;
- запуск приложения в браузере и мобильном WebView;
- REST-проверку пользователя;
- личные сообщения через модуль `im`.

Самописная часть отвечает за:

- заявки и строки заявки;
- версии;
- маршруты;
- задания согласующим;
- решения;
- назначения людей на роли;
- аудит.

## API

Первый экран `index.php` не подключает ядро Bitrix. Это повторяет устойчивый подход из UOR и снижает риск пустого 403 в мобильном приложении.

`api/bootstrap.php` также не подключает ядро и проверяет пользователя через REST.

Остальные API вызываются через:

```text
api/proxy.php -> api/worker.php -> requests.php/tasks.php/admin.php
```

Так mobile/browser обращаются к лёгкому endpoint, а ядро Bitrix подключается уже серверным worker-вызовом.

## Workflow MVP

Базовый маршрут хранится в `b_pr_route_rules`:

```text
warehouse -> profile_approver -> chief_engineer -> director -> registrar -> supply
```

Для каждого этапа создаются задания в `b_pr_tasks`. Исполнитель определяется по `b_pr_role_assignments` с учётом компании и площадки. Если исполнитель не найден, задание временно назначается администраторам из `PR_ADMIN_USER_IDS`, а событие пишется в аудит.

## Уведомления

Задание отправляет личное сообщение:

```text
FROM_USER_ID = PR_NOTIFY_FROM_USER_ID
TO_USER_ID = исполнитель
MESSAGE = "Вам пришёл бизнес-процесс..."
```

Стартовое значение `PR_NOTIFY_FROM_USER_ID = 1`.

## Вложения

Инициатор может прикрепить счета, коммерческие предложения и другие документы к черновику заявки или заявке на доработке. Файлы передаются через `api/proxy.php`, сохраняются в Bitrix через `CFile::SaveFile` и связываются с заявкой в `b_pr_attachments`.
